bashy SDLC Workflow

The scheduler or trigger is intentionally simple: it claims eligible work from any intake source, extracts a request pointer, calls bashy sdlc, watches state, and gates the next step. For public repositories, the recommended default is a local Loom control plane for private orchestration, with GitHub used for source control and external deployment handoff.

Smoke and accept

Trigger runs deterministic smoke checks and may assign QA. Rejection returns to the conductor.

↓
Optional UAT

User approval

Human approval is explicit and tied to the SDLC reference id.

Release
Trigger

Deploy

After required gates pass for the same reference id, the trigger invokes the configured deployment adapter.

↓
Done

Verify and close

Destination verification closes the reference. Failure routes rollback or another conductor loop.

Reject/fail loop: conductor failure, QA rejection, user rejection, smoke failure, deployment failure, or rollback need returns the same SDLC reference to the conductor with new evidence. The trigger does not decide the fix; it routes the next iteration.

Happy Path

  1. User, tool, or external system files a request in any configured intake source.
  2. Trigger claims eligible work, avoids duplicate processing, extracts a request pointer, and calls bashy sdlc.
  3. bashy sdlc creates a local lifecycle reference and invokes the configured conductor.
  4. The LLM conductor analyzes the request, runs a sprint, validates changes, commits work, and reports done.
  5. Trigger runs deterministic staging smoke checks, then optionally assigns QA. QA accepts or rejects.
  6. Trigger optionally requests human UAT approval. Approval is explicit.
  7. Trigger deploys through the configured destination adapter.
  8. Trigger verifies the destination, records evidence, and marks the SDLC reference resolved.

Role Boundaries

  • Trigger: deterministic scheduler/router; no architectural decisions.
  • SDLC: lifecycle state, request pointer, gate status, validation routing, deployment routing.
  • Conductor: LLM agent that owns planning, implementation, tests, and sprint loops.
  • QA/User: optional validation gates that can accept, reject, or request changes.
  • Deploy: starts only after configured gates pass for the same reference id, regardless of destination type.

Local Control Plane

  • Loom: local/private forge for mirrored issues, comments, CI state, and SDLC evidence.
  • GitHub: public source control plus integration with external deployment systems.
  • bashy sdlc: bridge from private intake to conductor, QA, approval, rollout, and resolution.
  • Security: keep subscription-backed agent execution and runtime logs off public GitHub Actions by default.

Recommended Additions

  • Claiming: mark intake items in-progress before invoking a conductor to prevent duplicate runs.
  • Idempotency: every action should be keyed by the SDLC reference id and safe to retry.
  • Evidence: store logs, test output, smoke results, approval notes, deploy ids, and destination checks.
  • Timeouts: conductor, QA, approval, and deployment should have explicit timeout states.
  • Adapters: intake and deployment should be pluggable; the lifecycle contract stays the same.
  • Rollback: failed destination verification should route either rollback or forward-fix as policy.

State Model

  • new -> request discovered but not claimed.
  • claimed -> trigger owns this cycle.
  • implementing -> conductor sprint is running.
  • ready_for_qa -> implementation reports done with evidence.
  • approved -> required QA/UAT gates passed.
  • deploying -> rollout in progress.
  • resolved or rejected or blocked -> terminal or waiting state.
