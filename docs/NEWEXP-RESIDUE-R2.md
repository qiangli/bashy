# new-exp Fixture Residue Analysis (R2)

This document clusters the remaining ~275 lines of failures observed in the `new-exp.tests` fixture from the Bash 5.3 test suite after the implementation of clusters 1, 3, and 4 from the original analysis.

## Candidate Issues

### 1. Parser Lenience for Numeric Prefixes in Command Substitutions (Residue of Original Cluster 5)
*   **Description:** `bashy` continues to incorrectly execute command substitutions that start with a numeric prefix (e.g., `$(11111...< file)`), instead of failing with a "command not found" error as expected in Bash 5.3. It appears the parser misidentifies the numeric prefix as a file descriptor or ignores it.
*   **Evidence:**
    ```
    < hi
    ---
    > ./new-exp2.sub: line 62: 1111111111111111111111: command not found
    ```
*   **Suspected Region:** `syntax/parser.go` (logic for identifying word tokens vs. redirections) and `interp/runner.go`.
*   **Fibonacci Estimate:** 5
*   **Status:** Persistent from original Cluster 5.

### 2. Anchored Substitution `${var/#/replacement}` Failure (Residue of Original Cluster 2)
*   **Description:** The anchored start substitution (`/#`) continues to fail in various contexts, particularly when trying to replace empty prefixes or matching specific characters.
*   **Evidence:**
    ```
    < abcd
    ---
    > zbcd
    (for test v=abcd; echo ${v/#a/z})
    ```
*   **Suspected Region:** `expand/param.go`.
*   **Fibonacci Estimate:** 3
*   **Status:** Persistent from original Cluster 2.

### 3. Positional Parameter Array Expansion with Empty Arguments (Residue of Original Cluster 6)
*   **Description:** Discrepancies persist in how `bashy` expands `@` and `*` when positional parameters are empty or contain null strings, leading to differences in the number of generated arguments (extra or missing `argv` entries).
*   **Evidence:**
    ```
    183a184
    > argv[1] = <>
    213d213
    < argv[1] = <>
    ```
*   **Suspected Region:** `expand/expand.go` and `expand/param.go`.
*   **Fibonacci Estimate:** 5
*   **Status:** Persistent from original Cluster 6.

### 4. Substring Error Message Discrepancy (Residue of Original Cluster 4)
*   **Description:** While negative substring offsets are now correctly validated and rejected (fixing the functional bug in original Cluster 4), the specific error message generated by `bashy` differs from Bash 5.3. `bashy` reports `@` as the expression while Bash 5.3 reports the evaluated negative value.
*   **Evidence:**
    ```
    < ./new-exp.tests: line 572: @: substring expression < 0
    ---
    > ./new-exp.tests: line 572: -2: substring expression < 0
    ```
*   **Suspected Region:** `expand/param.go` (error reporting in substring expansion).
*   **Fibonacci Estimate:** 2
*   **Status:** Residue of original Cluster 4 (functional fix landed).

### 5. Indirect Expansion combined with Substring or Pattern Operators (NEW)
*   **Description:** `bashy` fails with "bad substitution" when indirect expansion (`${!var}`) is combined with other operators like substring or pattern matching (e.g., `${!varname[@]@Q}`, `${!varname##aa}`). Bash 5.3 supports these combinations by evaluating the indirect reference first.
*   **Evidence:**
    ```
    ./new-exp13.sub: line 53: ${!varname[@]@Q}: bad substitution
    ./new-exp13.sub: line 71: ${!varname##aa}: bad substitution
    ```
*   **Suspected Region:** `expand/param.go` (handling of `syntax.ParamExp` when `Excl` is true).
*   **Fibonacci Estimate:** 8
*   **Status:** Genuinely New.

### 6. New Parameter Transformation Operators `@C`, `@a`, `@k` (NEW)
*   **Description:** `bashy` lacks support or has incorrect implementation for several parameter transformation operators introduced in Bash 5.1/5.3, specifically `@C` (capcase) and `@k` (key/value pairs), often resulting in "invalid expansion operator" or "bad substitution".
*   **Evidence:**
    ```
    bash: -c: line 1: invalid @ expansion operator `C`
    ./new-exp10.sub: line 153: ${V@}: bad substitution
    ```
*   **Suspected Region:** `expand/param.go` (switch case for transformation operators).
*   **Fibonacci Estimate:** 5
*   **Status:** Genuinely New.

### 7. Support for `&` in `${var/pat/repl}` Replacement Strings (NEW)
*   **Description:** `bashy` does not yet support the use of `&` in the replacement string to represent the matched portion of the pattern (a feature enabled by the `patsub_replacement` shopt in Bash 5.1+).
*   **Evidence:**
    ```
    < letx&yee
    ---
    > letxssyee
    ```
*   **Suspected Region:** `expand/param.go` (search and replace logic).
*   **Fibonacci Estimate:** 5
*   **Status:** Genuinely New.

### 8. Attribute Formatting and Sort Order in `declare -p` (NEW)
*   **Description:** `bashy`'s `declare -p` output uses a different attribute sort order and always includes explicit empty value assignments (`=''`), whereas Bash 5.3 may omit the value or use a different order.
*   **Evidence:**
    ```
    < declare -lr VAR1=''
    ---
    > declare -rl VAR1
    ```
*   **Suspected Region:** `interp/builtin.go` (`declare` implementation) and `interp/vars.go`.
*   **Fibonacci Estimate:** 3
*   **Status:** Genuinely New.

### 9. `[[` Keyword Recognition Issues in Subshells (NEW)
*   **Description:** In certain scripts, `bashy` fails to recognize `[[` as a keyword after toggling POSIX mode, reporting it as a "command not found". This suggests that the stricter POSIX parser might be incorrectly persisting or affecting bash-mode execution within the same process/runner.
*   **Evidence:**
    ```
    ./new-exp2.sub: line 48: [[: command not found
    ```
*   **Suspected Region:** `cmd/bashy/main.go` and `interp/runner.go` (interaction between POSIX mode and parser state).
*   **Fibonacci Estimate:** 8
*   **Status:** Genuinely New.

### 10. Prompt Expansion Environmental Discrepancy (ENVIRONMENTAL)
*   **Description:** Differences in prompt expansion (`\w`, `\W`) caused by `bashy` not abbreviating the home directory to `~` in this sandbox environment.
*   **Evidence:**
    ```
    < argv[1] = <^A[0]/Users/qiangli/.../tests$ >
    ---
    > argv[1] = <^A[0]~$ >
    ```
*   **Status:** Environmental.
